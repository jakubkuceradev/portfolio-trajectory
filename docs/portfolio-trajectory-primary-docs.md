# portfolio-trajectory Documentation

## 1. Project Overview

`portfolio-trajectory` is an open-source, browser-based lifecycle investing simulator for exploring probabilistic personal finance outcomes. The project focuses on modeling portfolio trajectories under historical market uncertainty using monthly block bootstrap sampling and user-defined cashflow events.

The application is intended to be polished enough for public hosting, portfolio/CV use, and demonstration to FIRE enthusiasts, investment hobbyists, average users planning financial goals, and potentially financial advisors or retirement planners.

The product is not intended to predict the future. Instead, it helps users compare financial strategies under historically grounded uncertainty and visually inspect the range of plausible outcomes.

## 2. Product Positioning

The strongest initial product identity is:

> An interactive lifecycle portfolio simulator with probabilistic cashflow events, historical block-bootstrap market sampling, and percentile-based outcome visualization.

The first version should favor a small number of well-executed features over a broad planning toolkit. Polish, interpretability, and visual clarity are core project goals.

## 3. Primary Use Case

The main question answered by the application is:

> Given my current age or time horizon, starting wealth, expected savings or withdrawals, uncertain future life events, and a selected portfolio strategy, what are the expected portfolio outcomes across different percentiles?

Users typically care about:

- Median outcome.
- Lower-bound outcomes such as the 5th or 10th percentile.
- Probability of ruin or failure.
- Spending sustainability.
- Terminal wealth.
- Comparison between portfolio strategies.
- How uncertain future events affect financial resilience.

The tool should support strategy evaluation rather than presenting a single deterministic forecast.

## 4. Target Audience

Primary audiences:

- FIRE and investment enthusiasts.
- Individuals trying to understand required savings or deposits to reach financial goals.
- People exploring retirement or long-horizon portfolio outcomes.
- Financial advisors and retirement planners who want to demonstrate strategy differences visually.

Secondary audiences:

- Developers reviewing the project as a technical portfolio piece.
- Quantitative finance hobbyists.
- Open-source contributors interested in simulation, visualization, or personal finance tools.

## 5. Design Principles

### 5.1 Polish Over Breadth

The first public version should be highly polished and constrained. The application should be impressive as a visual and interactive web product even if the feature set is narrow.

### 5.2 Correctness With Explicit Methodological Boundaries

Correctness is vital, but all forward-looking financial modeling is limited by assumptions about future returns, inflation, expenses, and user behavior. The tool should make assumptions visible, auditable, and easy to inspect.

### 5.3 Visual Intuition First

Users should understand the simulation through direct manipulation and clear visualization. The main chart should communicate uncertainty, percentile bands, and event impacts without requiring deep statistical knowledge.

### 5.4 Simulation Engine as a Core Asset

The simulation engine should be architected as a separable domain layer. Even if the first interface is a React web app, the simulation model should be testable, serializable, and independently understandable.

### 5.5 Future-Proof Data Model

The MVP should avoid complex features such as taxes, accounts, LETFs, conditionals, and event covariance, but the internal model should avoid blocking these future additions.

## 6. MVP Scope

The MVP should include:

### 6.1 Scenario Setup

Users can define:

- Time horizon in months or years.
- Starting portfolio value.
- Recurring monthly contribution or withdrawal.
- Asset allocation across stocks, bonds, and cash.
- Number of simulation runs.
- Bootstrap block length.
- Whether sampled blocks may wrap around the historical dataset.

### 6.2 Portfolio Model

The initial portfolio model uses three asset classes:

- Stocks.
- Bonds.
- Cash / risk-free asset.

The MVP strategy should support a fixed allocation with periodic rebalancing. Future strategy types should be anticipated by the architecture but not necessarily implemented immediately.

### 6.3 Historical Block Bootstrap Simulation

The simulation uses monthly historical return data. Each simulation path is generated by repeatedly sampling blocks of consecutive historical months until the desired horizon is filled.

The bootstrap model should:

- Use monthly time steps.
- Support configurable block length.
- Allow block length of one month, equivalent to ordinary bootstrap sampling.
- Optionally support wrap-around sampling.
- Sample asset returns and inflation as linked monthly vectors so the relationship between stocks, bonds, cash, and inflation is preserved within historical months.

### 6.4 Probabilistic Cashflow Events

Users can define one-time positive or negative cashflow events.

Each event may include uncertainty in:

- Whether it occurs.
- When it occurs.
- How large the amount is.

Initial supported distributions:

- Fixed.
- Uniform.
- Normal / Gaussian.

Example event types:

- House purchase.
- Large medical expense.
- Inheritance.
- Philanthropic gift.
- Other custom windfall or expense.

Goal events are not first-class entities in the MVP. Users evaluate goals through resulting statistics and visual outcomes.

### 6.5 Outcome Visualization

The primary visualization should be an interactive percentile fan chart showing portfolio value over time.

Default layers:

- 5th–95th percentile band.
- 10th–90th percentile band.
- 25th–75th percentile band.
- Median line.
- Optional mean line, visually deemphasized.
- Ruin threshold line.
- Event markers or event windows.

The chart should support hover or scrub interaction at any point in time.

### 6.6 Output Statistics

The MVP should calculate statistics for both nominal and real values where applicable.

Important statistics:

- Median terminal wealth.
- 5th and 10th percentile terminal wealth.
- Probability of ruin.
- Maximum drawdown.
- Longest underwater period.
- Time to recovery or time to break even.
- Distribution of final wealth.
- Possibly expected shortfall / CVaR for the bottom 5%.

### 6.7 Export

Users should be able to export:

- Scenario settings as JSON.
- Result summaries as JSON or CSV.

Shareable links may be added later. Links could represent either simulation settings or cached simulation results.

## 7. Non-MVP Scope

The following should be deferred until after the first polished version:

- User accounts.
- Server-side cached simulation runs.
- Taxes.
- Account types.
- Retirement income modeling.
- Salary growth modeling.
- Probabilistic income modeling.
- Event covariance matrices.
- Conditional events.
- LETF modeling.
- Complex asset products.
- Strategy scripting.
- Full financial advisor workflow.
- Goal objects distinct from cashflows.

These features may be considered in the data model and architecture, but should not expand the MVP surface area.

## 8. Timeline Model

The internal simulation unit is one month.

Reasons:

- Monthly data is compatible with long-horizon public investment datasets.
- It is granular enough for lifecycle planning.
- It avoids unsupported precision from daily modeling when the primary data is monthly.
- It supports block bootstrap windows of one to five years.

The default timeline should be represented as:

```text
month 0 -> month N
```

Age-based views may be layered on top later. For example, a user could specify current age, expected death age, retirement age, or probabilistic lifespan. These are not required for the first version.

Inflation should be tracked per simulation run because each bootstrap path contains its own sampled inflation history.

Values should be available in:

- Nominal dollars.
- Real dollars.

## 9. Market Data Model

Initial public datasets may include:

- Fama-French market factor data for US market returns.
- Fama-French or similar international/global equity data.
- Risk-free rate as a proxy for cash, short-term T-bills, or HYSA-like returns.
- Inflation/CPI from official public sources.
- Bond return datasets from public or open sources.

The application should prefer historical data going as far back as possible, but global equity data may constrain the practical start date, potentially around 1980 depending on dataset availability.

Where Fama-French returns are expressed as market excess returns, the total market return should be reconstructed by combining market excess return with the risk-free rate.

Dataset provenance should be visible in the UI or methodology documentation.

## 10. Bootstrap Methodology

Each simulation path is generated by sampling blocks of consecutive months from historical data.

Example:

1. Select a random start month from the historical dataset.
2. Copy `blockLength` consecutive months of linked monthly data.
3. Append the sampled block to the simulated future path.
4. Repeat until the simulated horizon is full.
5. Truncate the final block if it exceeds the simulation horizon.

Configurable settings:

- Number of runs.
- Horizon length.
- Block length.
- Wrap-around enabled or disabled.
- Asset universe.
- Real/nominal display mode.

A block length of one month produces ordinary bootstrap sampling. Longer blocks are intended to better preserve short-to-medium-term autocorrelation and historical sequence structure.

## 11. Portfolio Strategy Model

The MVP strategy is fixed target allocation with rebalancing.

Example:

```json
{
  "type": "fixed_allocation",
  "targetAllocation": {
    "stocks": 0.7,
    "bonds": 0.2,
    "cash": 0.1
  },
  "rebalance": {
    "type": "periodic",
    "frequencyMonths": 12
  }
}
```

Future strategies should fit into a generalized strategy abstraction.

Potential future strategies:

- Glide path.
- Age-based allocation.
- Threshold rebalancing.
- Contribution nudging toward target allocation.
- Lifecycle leverage.
- Conditional retirement-date shifting.
- Alternative asset allocation experiments.

Strategies should be serializable as JSON.

## 12. Event Model

An event is a probabilistic cashflow generator.

Conceptual schema:

```json
{
  "id": "event_house_purchase",
  "label": "House purchase",
  "type": "expense",
  "occurrenceProbability": 0.75,
  "date": {
    "distribution": "uniform",
    "minMonth": 60,
    "maxMonth": 120
  },
  "amount": {
    "distribution": "normal",
    "mean": 150000,
    "standardDeviation": 30000,
    "isRealValue": true
  }
}
```

Initial event design should answer:

- Does the event occur?
- In which month does it occur?
- What amount is applied?
- Is the amount positive or negative?
- Is the amount real or nominal?

For MVP, event occurrence, date, and amount can be independent. Correlations and conditionals are future features.

## 13. Simulation Order of Operations

The precise monthly order of operations should be made explicit and tested.

A reasonable default order:

1. Apply recurring contribution or withdrawal.
2. Apply event cashflows scheduled for the month.
3. Apply asset returns and inflation for the month.
4. Apply fees.
5. Rebalance if the strategy calls for it.
6. Record end-of-month portfolio state and statistics.

This ordering should be documented and encoded in tests because small changes can affect long-horizon results.

## 14. Ruin Semantics

The application should define ruin explicitly.

Possible MVP definition:

> Ruin occurs when portfolio wealth falls below or equal to zero after required withdrawals and events are applied.

Open design choice:

- Clamp wealth to zero after ruin.
- Allow wealth to become negative.
- Allow future contributions to revive the portfolio.

For a general-purpose lifecycle simulator, allowing revival after future contributions may be useful. For a retirement-only withdrawal model, clamping at zero may be more intuitive.

The MVP should choose one behavior and make it visible.

## 15. LETF Modeling

Leveraged ETF modeling is an advanced feature and should not be part of the MVP.

Future LETF modeling may use:

- Daily return data.
- Synthetic leverage applied to underlying index returns.
- Financing cost estimates.
- Expense ratios.
- Volatility drag.
- Regression against historical leveraged ETF behavior or index methodology data.

Because the base simulation model uses monthly data, LETF modeling from monthly returns would be approximate and should be treated as a specialized research extension.

## 16. Architecture Direction

The app should be a web application with a TypeScript/React frontend.

The core architecture should separate:

- UI components.
- Scenario state.
- Simulation engine.
- Data loading and validation.
- Statistics aggregation.
- Visualization transforms.
- Export/import logic.

Potential implementation paths:

### Option A: TypeScript Engine + Web Worker

Pros:

- Simplest development loop.
- Easier contributor onboarding.
- Shared types across UI and engine.
- Good enough for an MVP if optimized carefully.

Cons:

- May become slower for very large run counts.
- Numerical performance may be less impressive than Rust/WASM.

### Option B: Rust/WASM Engine + React Frontend

Pros:

- Strong performance story.
- Strong typing and memory control.
- Impressive CV/portfolio angle.
- Better fit for highly parallel simulation.

Cons:

- More complex tooling.
- More difficult debugging.
- More friction between frontend and engine types.
- Browser parallelism requires extra care.

A practical approach is to begin with a clean TypeScript engine running in a Web Worker and preserve a domain boundary that allows porting the engine to Rust/WASM later.

## 18. Visualization Design

The main interface should center around an interactive timeline/graph.

### Main View

- Percentile fan chart of portfolio value over time.
- Event timeline overlay.
- Scenario controls sidebar.
- Summary statistics panel.

### Supporting Views

- Terminal wealth distribution.
- Probability of ruin over time.
- Drawdown or underwater-period chart.
- Strategy comparison view later.

### Interaction Ideas

- Hover over any month to inspect percentile values.
- Toggle nominal vs real values.
- Toggle percentile bands.
- Click event markers to edit events.
- Drag event timing ranges directly on the timeline later.
- Hide advanced bootstrap settings by default.

## 19. Testing Strategy

Testing should be a visible part of the open-source project.

Recommended test categories:

- Deterministic simulation tests with fixed returns.
- Bootstrap sampling tests.
- Event sampling tests.
- Distribution tests.
- Rebalancing tests.
- Fee application tests.
- Real vs nominal conversion tests.
- Golden scenario tests.
- Performance benchmarks.

The simulation engine should have reproducible tests even if reproducibility is not a primary user-facing feature.
